# DTO·Entity 변환 방식 비교 (member-board-patterns 시리즈)

동일한 기능(회원 + 게시판 CRUD)을 네 가지 방식으로 나눈 프로젝트를 기준으로 정리했습니다.

| 폴더 | 단계 | 한 줄 요약 |
|------|------|-------------|
| `member-board-patterns` | 1. Setter | `new` 후 `setXxx()` 로 필드 복사 |
| `member-board-patterns1` | 2. 생성자 | `new Entity(...)`, `new ResponseDto(entity)` |
| `member-board-patterns2` | 3. Builder | Lombok `@Builder` 로 조립, DTO는 `from(entity)` 등 |
| `member-board-patterns3` | 4. Mapper | `UserMapper`, `BoardMapper` 에 변환 집중 |

---

## 1. DTO → Entity 변환이란?

**요청(폼·JSON)으로 받은 DTO**를 **DB에 저장할 JPA 엔티티**로 바꾸는 과정입니다.

| 단계 | 회원가입 예시 위치 | 패턴 |
|------|---------------------|------|
| 1 Setter | `UserService.register` | `User u = new User(); u.setEmail(...);` … |
| 2 생성자 | 동일 메서드 (patterns1) | `new User(email, passwordHash, nickname, now)` |
| 3 Builder | 동일 메서드 (patterns2) | `User.builder().email(...).build()` |
| 4 Mapper | `UserMapper.toNewEntity` 호출 (patterns3) | Service 는 `userMapper.toNewEntity(dto, encoded)` |

**주의:** 비밀번호처럼 “인코딩·검증”은 보통 **Service**에서 하고, Mapper 에는 **이미 인코딩된 값**만 넘기는 식으로 역할을 나눕니다 (patterns3).

---

## 2. Entity → DTO 변환이란?

**엔티티**를 그대로 화면/API에보내지 않고, **노출해도 되는 필드만** 담은 DTO로 옮기는 과정입니다.

| 단계 | 예시 | 패턴 |
|------|------|------|
| 1 Setter | `UserService` 의 private `toResponseDto` | `dto.setId(user.getId());` … |
| 2 생성자 | `UserResponseDto`, `BoardResponseDto` | `new UserResponseDto(user)` |
| 3 Builder | `UserResponseDto.from(user)` 내부 | `UserResponseDto.builder()...build()` |
| 4 Mapper | `UserMapper.toResponseDto`, `BoardMapper.toResponseDto` | 생성자 호출을 Mapper 한곳에 모음 |

이렇게 하면 **비밀번호 해시**처럼 응답에 넣지 않을 필드를 DTO 설계 단계에서 빼기 쉽습니다.

---

## 3. Service 계층 책임 (공통 vs 변환)

| 책임 | 내용 | 어느 단계에서 강조되는가 |
|------|------|-------------------------|
| 트랜잭션 | `@Transactional` 경계 | 모든 단계 공통 |
| 도메인 규칙 | 중복 이메일, 로그인 검증, 작성자만 수정/삭제 | 모든 단계 공통 |
| **객체 조립** | DTO ↔ Entity 필드 복사 | 1~3은 주로 Service 안, **4는 Mapper**로 이동 |

**4단계(Mapper)** 의 목적은 “Service 가 너무 길어지고, 변환 코드만 따로 테스트하고 싶다”는 **가독성·유지보수**입니다. 비즈니스 규칙 자체는 여전히 Service 가 갖습니다.

---

## 4. Mapper 를 분리하는 이유

- **단일 책임:** “HTTP/트랜잭션/규칙”과 “필드 복사”를 파일 단위로 나눔.
- **재사용:** 여러 Service 나 API 버전에서 같은 `toResponseDto` 를 호출하기 쉬움.
- **테스트:** Mapper 단위 테스트로 매핑 누락을 잡기 쉬움.

실무에서는 **MapStruct** 같은 코드 생성 매퍼를 쓰는 팀도 많고, **수동 Mapper 클래스**(patterns3)는 그 전 단계로 이해하면 됩니다.

---

## 5. Builder 를 쓰는 이유 (3단계)

- 필드가 많을 때 **`new` 나열 순서**를 외우지 않아도 됨.
- **선택 필드**·기본값을 다루기 편함.
- 읽을 때 **이름 있는 인자**처럼 읽혀서 리뷰가 수월함.

**JPA 영속 엔티티의 “부분 수정”**은 보통 Builder 로 새 인스턴스를 만들기보다, 이 프로젝트처럼 **`edit(...)` 같은 도메인 메서드**로 처리하는 경우가 많습니다 (관리 중인 엔티티와 충돌을 피함).

---

## 6. 생성자 vs Builder

| 구분 | 생성자 (2단계) | Builder (3단계) |
|------|----------------|-----------------|
| 의존성 | JDK 만으로 설명 가능 | Lombok 등 도구 |
| 가독성 | 인자가 3~4개 이하면 매우 좋음 | 인자가 많거나 선택값이 많을수록 유리 |
| 실무 | 도메인 객체·값 객체에서 흔함 | DTO/엔티티 생성 조립에서 흔함 |

둘 다 “**한 번에 유효한 상태**를 만든다”는 점에서는 Setter 나열보다 안전한 편입니다.

---

## 7. Setter 방식의 단점 (1단계)

- `set` 호출 **순서·누락** 실수가 나기 쉽고, 중간에 **반쯤만 세팅된 객체**가 생길 수 있음.
- 필드가 늘수록 Service 의 매핑 줄이 길어져 **리뷰·검색**이 불편해짐.
- **불변(immutable)** 을 유지하기 어렵고, 엔티티에 `@Setter` 를 열어두면 어디서든 바뀌어 **추적이 어려워질 수 있음**.

그래서 교육용·소규모에는 나오지만, 팀 프로젝트에서는 2~4단계 쪽으로 옮기는 경우가 많습니다.

---

## 8. 종합 비교표

| 방식 | 장점 | 단점 | 실무에서 |
|------|------|------|----------|
| **Setter** | 직관적, 초보자에게 읽기 쉬움 | 누락·순서 실수, 장황함, 불변과 맞물림 어려움 | 소규모·레거시·빠른 프로토타입 |
| **생성자** | 도구 없이 명확, 필수 값을 생성 시점에 고정 | 인자 많으면 `new` 가 길어짐 | 도메인 모델·간단 DTO 에 자주 사용 |
| **Builder** | 가독성, 선택 필드, 조립이 편함 | Lombok/JPA 조합 시 학습 곡선 | API DTO·복잡 조립에서 매우 흔함 |
| **Mapper 분리** | Service 가 얇아짐, 변환 테스트·재사용 | 클래스 수 증가, 팀 규칙 필요 | 중규모 이상, MapStruct 도입 전후 |

---

## 9. 포트폴리오·면접·학습 우선순위 (짧게)

**포트폴리오에 보여주기 좋은 구조**

- “요청/응답 DTO + 엔티티 분리” + “변환은 Mapper 또는 MapStruct” + “Service 는 규칙·트랜잭션” 정도를 한 문장으로 설명할 수 있으면 좋습니다.

**면접에서 Builder / Mapper 질문 예시 답변 방향**

- **Builder:** “필드가 많거나 선택값이 있을 때 가독성과 안전하게 한 번에 객체를 만들기 위해 쓴다. JPA에서 이미 관리 중인 엔티티의 부분 수정은 보통 setter 대신 도메인 메서드로 한다.”
- **Mapper:** “DTO 와 엔티티 변환을 Service 에서 빼서 책임을 나누고, 나중에 MapStruct 로 바꿔도 경계가 유지된다.”

**현업에서 자주 보는 조합**

- DTO 조립: **Builder** 또는 **정적 팩토리**  
- 엔티티: **생성자 + 도메인 메서드**  
- 변환 위치: **Mapper / MapStruct**

**지금 수준에서의 우선순위 제안**

1. **Entity 와 DTO 를 왜 나누는지** (노출·검증·스키마 독립)  
2. **2단계 생성자**로 “반쯤 세팅” 문제 이해  
3. **3단계 Builder**로 실무 스타일에 가깝게 작성  
4. **4단계 Mapper**로 계층 역할 말하기 연습 → 필요 시 MapStruct 로 확장
