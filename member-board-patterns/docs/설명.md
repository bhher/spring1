# member-board-patterns — 학습용 설명서

회원·게시판 CRUD를 통해 **DTO ↔ Entity 변환 방식**을 단계별로 비교하기 위한 Spring Boot 프로젝트입니다.  
현재 저장소에는 **1단계: Setter 방식**이 구현되어 있습니다.

### 단계별 복사본 (같은 기능, 변환 방식만 다름)

| 폴더 | 단계 |
|------|------|
| `member-board-patterns` | 1단계 Setter |
| `member-board-patterns1` | 2단계 생성자 |
| `member-board-patterns2` | 3단계 Lombok Builder |
| `member-board-patterns3` | 4단계 Mapper 분리 |

단계 간 **개념·장단점·실무** 비교는 [비교.md](./비교.md) 를 참고하세요.

## 구현된 기능 (1단계)

### 회원

| 기능 | HTTP | 경로 |
|------|------|------|
| 회원가입 폼/처리 | GET/POST | `/signup` |
| 로그인 폼/처리 | GET/POST | `/login` (쿼리 `redirect` 지원) |
| 로그아웃 | POST | `/logout` |
| 회원정보 조회 | GET | `/me` (로그인 필요) |

로그인 상태는 **세션**에 `loginUserId`(Long) 저장. 보호 URL은 `LoginCheckInterceptor`로 처리합니다.

### 게시판

| 기능 | HTTP | 경로 |
|------|------|------|
| 목록 | GET | `/boards` |
| 상세 | GET | `/boards/{id}` |
| 작성 폼 | GET | `/boards/new` |
| 작성 처리 | POST | `/boards/create` |
| 수정 폼/처리 | GET/POST | `/boards/{id}/edit` |
| 삭제 | POST | `/boards/{id}/delete` |

목록·상세는 비로그인 접근 가능. 작성·수정·삭제는 로그인 필요이며, 수정·삭제는 **작성자 본인**만 가능합니다.

> POST 경로를 `/boards/create`로 둔 이유: 목록 `GET /boards`와 충돌하지 않게 하기 위함입니다.

## 1단계: Setter 방식이란?

**화면·요청용 객체(DTO)**와 **DB에 저장되는 객체(Entity)**를 서로 옮길 때,  
`new`로 빈 객체를 만든 뒤 **`setXxx(...)`로 필드를 하나씩 채우는 방식**입니다.

### DTO → Entity (회원가입 예)

`UserService.register`에서 폼이 넘어온 `UserRegisterDto`를 `User` 엔티티로 옮깁니다.

- `User user = new User();`
- `user.setEmail(dto.getEmail());`
- 비밀번호는 평문을 DB에 넣지 않고 `passwordEncoder.encode(...)` 후 `setPasswordHash`
- `user.setNickname(...)`, `user.setCreatedAt(...)`

코드 위치: `UserService.java`의 `register` 메서드.

### Entity → DTO (프로필·목록 응답 예)

엔티티를 그대로 화면에 넘기지 않고, 응답 전용 `UserResponseDto` / `BoardResponseDto`를 만든 뒤 setter로 채웁니다.

- `UserService.toResponseDto(User user)` — `UserResponseDto dto = new UserResponseDto();` 후 `setId`, `setEmail` 등
- `BoardService.toResponseDto(Board board)` — 작성자 닉네임 등 노출할 필드만 복사

이렇게 하면 **비밀번호 해시**처럼 API·화면에내면 안 되는 필드는 DTO에 넣지 않아서 막기 쉽습니다.

### 게시글 수정 시 DTO → Entity

`BoardService.update`에서는 이미 DB에 있는 `Board`를 조회한 뒤, DTO 값으로 **`setTitle`, `setContent`, `setUpdatedAt`**만 갱신합니다. (새 엔티티를 만들지 않음)

## 계층별 역할 (이 프로젝트 기준)

| 계층 | 역할 |
|------|------|
| **Controller** | HTTP 요청/응답, Thymeleaf 모델 바인딩, 검증 오류 시 다시 폼으로 이동. 가능한 한 가볍게 유지. |
| **Service** | 트랜잭션 경계, 도메인 규칙(중복 이메일, 로그인 검증, 작성자만 수정/삭제), **DTO ↔ Entity 변환(1단계에서는 Setter)**. |
| **Repository** | JPA로 엔티티 저장·조회. `join fetch`로 N+1 완화 예시 포함. |
| **DTO** | 입력(`UserRegisterDto`, `UserLoginDto`, `BoardWriteDto`) / 출력(`UserResponseDto`, `BoardResponseDto`) 분리. |
| **Entity** | 테이블 매핑, 연관관계(`Board` → `User` 작성자). |

## Setter 방식 비교표 (학습용)

| 구분 | 장점 | 단점 | 실무에서 |
|------|------|------|----------|
| **Setter** | 초보자가 읽기 쉽고, 디버거로 한 줄씩 추적하기 좋음 | 필드가 많아지면 매핑 코드가 길어지고, **set 호출 순서·누락** 실수가 나기 쉬움; 불변(immutable) 객체와는 잘 안 맞음 | 교육·소규모 CRUD에서는 볼 수 있으나, 팀 규모가 커지면 Mapper/Builder 등으로 정리하는 편이 많음 |

---

## 실무에서는 (1단계 기준)

- **소규모 과제·입문 예제**에서는 Setter 매핑도 이해용으로는 충분합니다.
- **필드가 늘거나** 같은 엔티티에서 **여러 종류의 DTO**로 나가야 하면, Service 안의 setter 나열이 반복·지저분해지므로 이후 단계에서 **생성자 / Builder / Mapper 분리**로 옮기는 경우가 많습니다.

## 다음 단계 예고 (같은 프로젝트 로드맵)

| 단계 | 방식 | 비교 포인트 |
|------|------|-------------|
| 2 | 생성자 | 필수 값을 생성 시점에 고정, Setter 남용 감소 |
| 3 | Builder | 가독성·선택 필드·불변에 유리 |
| 4 | Mapper 클래스 | 변환 로직을 Service 밖으로 빼 **단일 책임**·테스트 용이 |

2~4단계 코드는 이 문서 작성 시점에는 **미구현**이며, 동일 URL·기능을 유지한 채 변환 부분만 바꿔 비교하는 것을 권장합니다.

---

## 한 줄 정리

- **1단계**: 변환은 전부 **Service의 `new` + setter**로 처리했습니다.  
- 동작을 먼저 익힌 뒤, 2단계부터 같은 메서드 시그니처를 유지하며 **변환 코드만** 바꿔 보면 학습 효과가 큽니다.

질문이나 2단계(생성자) 적용을 원하면 이어서 요청해 주세요.
